## [→ ТВОЙ ХОД] Когда приедут ноды

### Шаг 1. Прошить

Открой **https://hleserg.github.io/meshcore-node-fw**, выбери плату и тип прошивки.

**V4 — кликом:** Chrome или Edge на десктопе, воткни плату в USB, жми
«Прошить V4 через браузер», выбери порт.
*Успех:* прогресс до 100%, «Installation complete».

**T114 — файлом:** скачай `.uf2`, воткни плату, **быстро нажми `RST` два раза**,
появится диск `T114BOOT`, перетащи файл.
*Успех:* диск исчезает сам, плата перезагружается.

### Шаг 2. Стартовый лог — сверка пинов

Не отключая USB, открой порт **платы** терминалом на 115200 и нажми `RST`:

```
=== MeshCore companion :: transport auto-detect ===
firmware  : v0.2.0-… (…)
DETECT=GPIO33 (P1 pin 8) = LOW -> ble | UART RX=GPIO9 (P1 pin 12), TX=GPIO10 (P1 pin 13)
detect    : arduino 33  -> P1.01
detect    : LOW (nothing wired)
transport : BLE
note      : pull the detect pin HIGH and reboot to use the wire
==================================================
```

**Это и есть сверка «силк ≠ Arduino-пин».** Если на T114 видишь `P1.01` для detect и
`P0.09`/`P0.10` для UART — распиновка та, что заявлена, можно паять.
Если что-то другое — стоп, пришли мне вывод, дальше не иди.

### Шаг 3. Проверить оба режима

**Без провода** (detect висит в воздухе → LOW):
- V4 поднимает WiFi-точку `MeshCore-<имя-ноды>`. Подключись телефоном, IP видно на
  экранчике. Логгер — на `192.168.4.1:5000`.
- T114 виден по Bluetooth, цепляется приложением MeshCore.

**С проводом** (detect подтянут к 3.3 В → HIGH), провода:

**T114 — всё на хедере P1:**

| Сигнал | Позиция | GPIO |
|---|---|---|
| GND | P1 пин 4 | — |
| DETECT ← 3.3 В | P1 пин 8 | `GPIO33` (P1.01) |
| RX ← TX адаптера | P1 пин 12 | `GPIO9` (P0.09) |
| TX → RX адаптера | P1 пин 13 | `GPIO10` (P0.10) |

**V4 — всё на хедере J2:**

| Сигнал | Позиция | GPIO |
|---|---|---|
| GND | J2 пин 1 | — |
| DETECT ← 3.3 В | J2 пин 12 | `GPIO33` |
| RX ← TX адаптера | J2 пин 13 | `GPIO47` |
| TX → RX адаптера | J2 пин 14 | `GPIO48` |

Перезагрузи ноду (смена режима только через ребут), проверь лог — должно быть
`HIGH` и `HARDWARE UART`. Потом:

```bash
pip install meshcore
python scripts/selftest.py COM5          # порт АДАПТЕРА, не платы
```

*Успех:* печатается self-info и `OK - the companion protocol is alive on the hardware UART.`

### Шаг 4. Выставить радио

Обязательно, иначе ноды не услышат друг друга:

```
meshcli -s COM5 set radio 869.525,250,11,5
```

## Если не так → почему → что делать

| Симптом | Причина | Фикс |
|---|---|---|
| В логе detect-пин не тот, что заявлен | Силк Heltec не совпал с Arduino-пином на твоей ревизии | Стоп. Пришли вывод лога — поправлю `TRANSPORT_DETECT_PIN` |
| Стартового лога нет вообще | Терминал не на том порту либо порт занят | Порт **платы**, не адаптера. Закрой другие терминалы. Нажми `RST` при открытом терминале |
| **Нода не видна телефону** (нет WiFi-точки / не видно по BLE) | detect прочитался HIGH — на пине наводка или провод забыт | Вынь провод с detect-пина и **перезагрузи ноду**. Режим выбирается только на старте |
| **Нода молчит по проводу** | detect прочитался LOW — нода ушла в радио-режим | Подтяни detect к 3.3 В и перезагрузи. Проверь по логу, что стало `HIGH` |
| Лог говорит `HIGH`, а self-info молчит | 9 из 10 — перепутаны RX/TX | Поменяй местами два провода. Крест-накрест |
| То же, провода верные | Нет общей земли | Соедини GND платы и GND адаптера |
| То же, земля есть | Адаптер на 5 В | Нужен 3.3 В. На 5 В плата может быть уже повреждена |
| T114 тупит пару секунд после первой прошивки | Однократная перезапись UICR (освобождение NFC-пинов) и ребут | Так и задумано, подожди 2–3 с |
| T114 после смены прошивки: не поднимается ФС, слетели настройки | Остатки предыдущей прошивки | Разово залей erase-`.uf2` из [MeshCore/bin](https://github.com/meshcore-dev/MeshCore/tree/main/bin), потом наш образ |
| V4: кнопка «Прошить» серая | Firefox или Safari — там нет Web Serial | Только Chrome или Edge, десктоп |
| V4: кнопка есть, порт не открывается | Страница не по HTTPS либо порт занят | Открывай `https://hleserg.github.io/…`, закрой терминалы |
| Логгер не достучался до `192.168.4.1` | Android увёл трафик в LTE — точка без интернета | Отключи мобильные данные на время прогона |
| Ноды не слышат друг друга | Радио не выставлено после прошивки | `meshcli -s COM5 set radio 869.525,250,11,5` на **всех** нодах |
| Страница отдаёт старый образ | Кэш браузера | `Ctrl+Shift+R`. Версия видна в подвале страницы |
| Pages не задеплоился (404) | Job `Deploy field hub` упал | `gh run list` → последний ран на `main` |

## Само-ревью: что проверил, а что предположил

`TODO`/`FIXME` — ноль. Секретов и токенов — ноль. Ни одного радио-параметра мы сами не
задаём. Но четыре вещи честнее сказать прямо:

**1. Режим WiFi я поменял относительно стока — сознательно.** Бриф говорит «WiFi-точка,
поведение как в `companion_radio_wifi`», но это два разных режима: сток делает
`WiFi.begin(SSID, PWD)` — станцию, которая подключается к чужой сети и адрес
`192.168.4.1` дать не может в принципе. `softAP` в MeshCore есть только для OTA. На
фесте роутера нет, а весь полевой процесс завязан на `192.168.4.1:5000`. Поэтому здесь
точка доступа. Если это не то — скажи, верну станцию, но тогда нужен SSID/пароль в
сборке и рабочий процесс поменяется.

**2. Точка доступа открытая.** Без пароля, как в инструкции. Задаётся флагом
`-D WIFI_AP_PASSWORD='"..."'`, если нужен.

**3. Девственная плата поднимается на 869.618 МГц** и сразу даёт advert
(`ENABLE_ADVERT_ON_BOOT=1`) — унаследовано из `[arduino_base]`. То есть до того, как ты
выставишь радио, нода уже в эфире на другой частоте.

**4. `ENABLE_PRIVATE_KEY_IMPORT/EXPORT=1`** унаследованы, апстрим помечает их как менее
безопасные. Оставил как в стоке.

**Чего не делал вообще:** не трогал железо. Не проверял, что detect-пин физически
выведен на тот пад, который я назвал, — это и есть смысл шага 2.
