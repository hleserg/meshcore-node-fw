# meshcore-node-fw

Кастомные companion-прошивки [MeshCore](https://github.com/meshcore-dev/MeshCore)
для двух плат Heltec, у которых **companion-протокол выведен на аппаратный UART**,
а не на USB-CDC или BLE. Нода подключается по проводу — например, к Flipper Zero,
который пишет лог.

- **WiFi и BLE отключены.** В `examples/companion_radio/main.cpp` интерфейс
  выбирается по приоритету `WIFI_SSID` → `BLE_PIN_CODE` → `SERIAL_RX` → `Serial` (USB).
  Ни `WIFI_SSID`, ни `BLE_PIN_CODE` в этих сборках не определены, поэтому побеждает
  аппаратный serial.
- **Параметры радио не захардкожены.** Частота, BW, SF, CR задаются в рантайме через
  companion-протокол; значения в сборке — только заводские дефолты MeshCore.
- **Собирается в облаке.** GitHub Actions делает оба образа, кладёт их в Releases
  и публикует страницу-флешер. PlatformIO на своей машине ставить не нужно.

## Полевой хаб

**→ [hleserg.github.io/meshcore-node-fw](https://hleserg.github.io/meshcore-node-fw)**

Одна ссылка на всё: прошить ноду, скачать логгер `meshlog.py`, инструкции.
**Инструкции и логгер работают оффлайн** — страницу можно добавить на экран телефона
и читать в поле без интернета. Прошивка интернет требует (Web Serial + CDN).

Первый раз с платой в руках — [HANDOFF.md](HANDOFF.md): пошагово, что нажать, как
выглядит успех и таблица «симптом → причина → фикс».

| Плата | Чип | Как шьётся |
|---|---|---|
| Heltec WiFi LoRa 32 **V4** | ESP32-S3 | **Кликом в браузере** (Chrome/Edge, Web Serial) |
| Heltec Mesh Node **T114** | nRF52840 | **Скачать `.uf2`** → двойной тап `RST` → перетащить на диск |

Готовые файлы также лежат в [Releases](../../releases): `v4-factory.bin` и
`t114-companion.uf2`.

## Подключение по UART

3.3 В, 115200 8N1.

| T114 (хедер P1) | | V4 (хедер J3) | |
|---|---|---|---|
| `GPIO9` (P0.09) — RX | ← TX адаптера | `GPIO47` — RX | ← TX адаптера |
| `GPIO10` (P0.10) — TX | → RX адаптера | `GPIO48` — TX | → RX адаптера |
| `GND` | ↔ GND | `GND` | ↔ GND |

Проверить, что нода отвечает:

```bash
pip install meshcore
python scripts/selftest.py COM5        # или /dev/ttyUSB0
```

### Стартовый баннер

При старте прошивка печатает в **USB-CDC** (не в companion-UART) назначенные пины —
чтобы сверить распиновку на живой плате, не разбирая исходники. Откройте USB-порт платы
терминалом на 115200 и нажмите `RST`:

```
=== MeshCore companion :: hardware UART ===
firmware  : v0.1.0-07a3ca9e (23-Jul-2026)
port      : Serial2 (NRF_UARTE)
RX        : arduino 9 -> P0.09
TX        : arduino 10 -> P0.10
baud      : 115200 8N1
wiring    : board RX <- adapter TX, board TX -> adapter RX, GND <-> GND
note      : companion protocol is on the UART above, NOT on this USB port
==========================================
```

На nRF52 печатается именно **порт-пин** (`P0.09`), разрешённый через `g_ADigitalPinMap`,
а не тот номер, что мы передали — иначе это была бы не проверка, а повтор предположения.

## Какой параметр на каком слое

Проверено по исходникам MeshCore, а не по документации:

| Параметр | Слой | Чем подтверждается |
|---|---|---|
| Частота, BW, SF, CR | **runtime** | `CMD_SET_RADIO_PARAMS = 11` |
| Мощность передатчика | **runtime** | `CMD_SET_RADIO_TX_POWER = 12` |
| **Path hash mode** | **runtime** | `CMD_SET_PATH_HASH_MODE = 61`, поле `NodePrefs.path_hash_mode`, персистится и отдаётся в self-info |
| Имя ноды, координаты, каналы, контакты | **runtime** | отдельные `CMD_*` |
| `PATH_HASH_SIZE = 1` | compile-time | `#define` в `src/MeshCore.h` |
| Пины UART, выбор интерфейса | compile-time | build-флаги этих сборок |

**Радио не зашито в прошивку.** `LORA_FREQ`/`LORA_BW`/`LORA_SF`/`LORA_CR` в build-флагах —
это только заводские дефолты. Порядок в `MyMesh::begin()` такой:

1. `_prefs.freq = LORA_FREQ; …` — засеять дефолтами (`MyMesh.cpp:874`);
2. `_store->loadPrefs(_prefs, …)` — перетереть сохранённым с флешки;
3. `radio_driver.setParams(_prefs.freq, _prefs.bw, _prefs.sf, _prefs.cr)` — применить
   к радио (`MyMesh.cpp:964`).

То есть build-флаги влияют только на девственную файловую систему; дальше всё решает
рантайм.

**Про `PATH_HASH_SIZE`.** Да, это compile-time и да, он влияет на формат пакета — но это
константа wire-формата протокола V1, а не настраиваемый профиль. Во-первых, это голый
`#define` без `#ifndef`, поэтому `-D PATH_HASH_SIZE=…` его даже не переопределит:
заголовок переопределит значение обратно. Во-вторых, менять его в одной ноде значит
отвалиться от всей остальной сети MeshCore.

**Следствие: предсобранные варианты под «профили сети» не нужны** — вшивать в прошивку
нечего, всё сетевое ставится по проводу в рантайме. Поэтому на странице-флешере нет
выпадающего списка профилей: он был бы обманом.

⚠️ **Но дефолт не нейтральный.** Из апстримного `[arduino_base]` наследуются
`LORA_FREQ=869.618`, `LORA_BW=62.5`, `LORA_SF=8`, `LORA_TX_POWER=22` и
`ENABLE_ADVERT_ON_BOOT=1`. На чистой файловой системе (первая прошивка или после erase)
нода стартует именно на **869.618 МГц** и сразу даёт advert в эфир — до любой настройки.
«Не захардкожено» здесь значит «не залочено», а не «частоты нет».

⚠️ Также унаследованы `ENABLE_PRIVATE_KEY_IMPORT=1` / `ENABLE_PRIVATE_KEY_EXPORT=1` —
апстрим помечает их как «comment these out for more secure firmware». Приватный ключ
ноды выгружается командой companion-протокола; у нас она доступна только по проводу,
то есть требует физического доступа к пинам.

## Ограничения

- **V4 из браузера — только Chrome или Edge на десктопе.** Прошивка использует
  Web Serial API; в Firefox и Safari его нет. Страница должна открываться по HTTPS
  (или с `localhost`) — иначе браузер не даст доступ к порту.
- **T114 из браузера не шьётся вообще.** nRF52840 не отдаёт себя по Web Serial —
  единственный путь — UF2-перетаскивание. Это ограничение платформы, а не страницы.
- **Странное поведение T114 после смены прошивки** (не поднимается файловая система,
  сбрасываются настройки) лечится разовым прогоном erase-прошивки `.uf2` из
  [`MeshCore/bin`](https://github.com/meshcore-dev/MeshCore/tree/main/bin),
  после чего можно шить обычный образ.
- **`GPIO9`/`GPIO10` на nRF52840 — это NFC-пины.** Прошивка собирается с
  `-D CONFIG_NFCT_PINS_AS_GPIOS`, поэтому при первом старте она однократно
  перепишет UICR и перезагрузит плату. Это нормально.

## Как это собрано

- `external/MeshCore` — сабмодуль апстрима, зафиксированный на теге.
- `patches/` — минимальный патч на `examples/companion_radio/main.cpp`, добавляющий
  ветку hardware-serial для nRF52 и выбор UART-периферии для ESP32.
- `pio/meshcore-node-fw.ini` — два PlatformIO-окружения; копируется в
  `external/MeshCore/platformio.local.ini`, который апстрим уже подхватывает через
  `extra_configs`.
- `scripts/` — `prepare.py` (накатить патчи), `build.py` (собрать и разложить
  артефакты), `selftest.py` (проверить UART).

Подробности сборки, флаги и обоснование выбора пинов — в [BUILD.md](BUILD.md).
