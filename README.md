# meshcore-node-fw

Три образа прошивок [MeshCore](https://github.com/meshcore-dev/MeshCore) для нод Heltec,
собираемые в облаке. Companion-сборки **сами выбирают транспорт на старте**: подключён
провод от флиппера — работают по проводу, не подключён — поднимают WiFi-точку (V4) или
BLE (T114).

**→ [Страница прошивки](https://hleserg.github.io/meshcore-node-fw)**

Инструкции и логгер живут отдельно: **[hleserg.github.io/flipperMeshCoreConfig](https://hleserg.github.io/flipperMeshCoreConfig)**

## Три образа

| Файл | Плата | Что это |
|---|---|---|
| `t114-companion.uf2` | T114 (nRF52840) | companion + автодетект: UART ↔ BLE |
| `v4-companion-factory.bin` | V4 (ESP32-S3) | companion + автодетект: UART ↔ WiFi-точка |
| `t114-repeater.uf2` | T114 (nRF52840) | **стоковый** repeater, без патчей |

Лежат в [Releases](../../releases) и на странице прошивки.

## Кто что шьёт

| | Ноды | Прошивки |
|---|---|---|
| **Сергей** | `BASE`, `REP1`, `REP2` + `ROVER-S`, `POCKET-S` | 3 × repeater, 2 × companion |
| **Марк** | `REP3` + `ROVER-M` (V4), `POCKET-M` (T114) | 1 × repeater, 2 × companion |

## Как работает автодетект

На старте, **до** инициализации любого транспорта, прошивка читает detect-GPIO
(вход с внутренним pull-down):

| Уровень | Что значит | Транспорт |
|---|---|---|
| **HIGH** | провод от флиппера подключён | companion-протокол на аппаратном UART. WiFi и BLE не поднимаются |
| **LOW** | ничего не подключено | V4 → WiFi-точка + TCP 5000 · T114 → BLE |

Решение принимается **один раз**. Смена режима — перезагрузка ноды, горячего
переключения нет. Pull-down означает, что «ничего не подключено» — состояние покоя:
неподключённый пин читается как LOW, и нода поднимается по радио, как стоковая.

### Распиновка

3.3 В, UART 115200 8N1. RX платы идёт на TX адаптера.

| | T114 (хедер **P1**) | V4 |
|---|---|---|
| **DETECT** | `GPIO13` = P0.13 | `GPIO33` |
| UART RX | `GPIO9` = P0.09 ← TX адаптера | `GPIO47` ← TX адаптера |
| UART TX | `GPIO10` = P0.10 → RX адаптера | `GPIO48` → RX адаптера |
| GND | P1·4 | J2·1 |

Пины T114 взяты из официального датасheet (Rev 1.0, §2.2 «Pin Definition»), где прямо
написано `0.09 → GPIO9, UART1_RX`, и сверены с `variants/heltec_t114/variant.h`.
`g_ADigitalPinMap` на этой плате — тождественное отображение, поэтому номер
Arduino-пина, маркировка Heltec и порт-пин nRF здесь совпадают. DETECT выбран на том же
хедере P1, где уже паяется UART — чтобы не тянуть провод через плату.

**Проверить, что пины угаданы верно:** воткни USB, открой порт платы терминалом на
115200 и нажми `RST`. Прошивка печатает уровень detect, выбранный транспорт и все
назначенные пины — см. [HANDOFF.md](HANDOFF.md).

## Какой параметр на каком слое

Проверено по исходникам MeshCore, а не по документации:

| Параметр | Слой | Чем подтверждается |
|---|---|---|
| Частота, BW, SF, CR | **runtime** | `CMD_SET_RADIO_PARAMS = 11` |
| Мощность передатчика | **runtime** | `CMD_SET_RADIO_TX_POWER = 12` |
| **Path hash mode** | **runtime** | `CMD_SET_PATH_HASH_MODE = 61`, поле `NodePrefs.path_hash_mode` |
| `PATH_HASH_SIZE = 1` | compile-time | `#define` в `src/MeshCore.h` |
| Пины UART и detect | compile-time | build-флаги этих сборок |

**Радио не зашито.** В `MyMesh::begin()`: `_prefs.freq = LORA_FREQ` засевает дефолты
(`MyMesh.cpp:874`) → `loadPrefs()` перетирает сохранённым → `radio_driver.setParams(...)`
применяет (`:964`). Build-флаги влияют только на девственную файловую систему.

**Про `PATH_HASH_SIZE`.** Да, compile-time, и да, влияет на формат пакета — но это
константа wire-формата протокола V1, а не профиль сети. Это голый `#define` без
`#ifndef`, поэтому `-D PATH_HASH_SIZE=…` его даже не переопределит. И менять его в одной
ноде значит отвалиться от всей остальной сети MeshCore.

**Следствие: предсобранные варианты под «профили сети» не нужны** — вшивать в прошивку
нечего. Поэтому на странице нет выпадающего списка профилей.

⚠️ Но дефолт не нейтральный: из `[arduino_base]` наследуются `LORA_FREQ=869.618`,
`BW=62.5`, `SF=8`, `TX=22` и `ENABLE_ADVERT_ON_BOOT=1`. На чистой файловой системе нода
стартует именно на 869.618 и сразу даёт advert. Поэтому после прошивки радио надо
выставить — команда есть на странице прошивки.

## Ограничения

- **V4 из браузера — только Chrome или Edge на десктопе**, по HTTPS. Web Serial в
  Firefox и Safari отсутствует.
- **T114 из браузера не шьётся вообще** — nRF52840 не отдаёт себя по Web Serial.
  Только UF2-перетаскивание.
- **Странное поведение T114 после смены прошивки** лечится разовым прогоном
  erase-`.uf2` из [`MeshCore/bin`](https://github.com/meshcore-dev/MeshCore/tree/main/bin).
- **`GPIO9`/`GPIO10` на nRF52840 — NFC-пины.** Сборка идёт с
  `-D CONFIG_NFCT_PINS_AS_GPIOS`, поэтому при первом старте прошивка однократно
  перепишет UICR и перезагрузит плату.
- **WiFi-точка V4 открытая** (без пароля) и без интернета. Пароль задаётся флагом
  `-D WIFI_AP_PASSWORD='"..."'`, если нужен.

Подробности сборки, флаги и патч — в [BUILD.md](BUILD.md).
